Making a Copy of the Training Materials

If you'd like to make a copy of the course materials for later study, run the following command from a terminal on your gateway node's desktop:

cd ~/training_materials
sudo zip -r /opt/tomcat7/webapps/ROOT/training_materials.zip .

You should then be able to download them with a browser on your local machine, by using the following URL (replace ipaddress with the public IP address of your gateway node, as provided by your instructor):

http://ipaddress:8080/training_materials.zip

Creating Topics

Examples of creating topics are in many of our exercises. You may wish to use retention.bytes and segment.bytes to prevent your Kafka brokers in this environment from running out of disk space:

kafka-topics \
    --zookeeper $ZK_CONNECT_STRING \
    --create \
    --topic newcustomers \
    --partitions 3 \
    --replication-factor 3 \
    --config retention.bytes=1073742000 \
    --config segment.bytes=1073742000

An explanation of retention.bytes, segment.bytes and examples of using retention.ms and segment.ms are found in:

Demonstrating the Effect of the Message Retention Period

Viewing Partition Assignments for a Topic

The kafka-topics tool shows broker assignments for the topic:

kafka-topics --zookeeper $ZK_CONNECT_STRING \
             --describe \
             --topic YOUR_TOPIC_HERE

Example output:

Topic:hello    PartitionCount:2    ReplicationFactor:3    Configs:min.insync.replicas=2
Topic: hello    Partition: 0    Leader: 35    Replicas: 35,34,41    Isr: 35,34,41
Topic: hello    Partition: 1    Leader: 35    Replicas: 35,32,40    Isr: 35,32,40

Viewing out of sync replicas and under-replicated partitions

Use the kafka-topics command with the --unavailable-partitions or the --under-replicated-partitions switches.

Increase Replication Factor for a Topic

This is a multi-step process, based loosely on instructions found here:

Kafka Reassign Partitions

Step 1: Describe Topic

kafka-topics \
     --zookeeper $ZK_CONNECT_STRING \
     --describe passwordreset

Note the ID of the broker that is the leader for this topic's (only) partition. For the sake of example, we will assume that is broker ID #7.

Step 2: Obtain Another Broker's ID

You can use the zookeeper-client command or the kafka-api-versions command (See below) to obtain broker IDs.

zookeeper-client \
    -server worker1.example.com:2181 \
      get /kafka/brokers/ids

Note the ID of a broker other than the one in the first step. For the sake of this example, we will assume that is broker ID #11.

Step 3: Create a JSON "mappings" file

{
     "version":1,
      "partitions":[
            {"topic":"passwordreset","partition":0,"replicas":[7, 11]}
             ]
}

It does not matter what the file is called, but for the sake of example, we will assume that it's called add-replicas.json.

Step 4: Run the kafka-reassign-partitions Command

kafka-reassign-partitions \
    --zookeeper $ZK_CONNECT_STRING \
    --reassignment-json-file add-replicas.json \
    --execute

If you describe the topic again, you should find that the replication factor is now 2.

Rebalancer Tool

The rebalancer tool can be used to reassign leaders, followers for topic partitions. Additionally, (as shown above) it can be used to increase the replication factor for a topic.

Broker ID to hostname mapping

You can find hostnames for the leader/followers by using the kafka-api-versions command:

kafka-broker-api-versions --bootstrap-server $BOOTSTRAP_SERVERS | grep id

Check offsets for a topic

  • Check earliest and latest offsets for a topic:

The numbers specified by --time -1 and -2 mean "latest offset" and "earliest offset", respectively

kafka-run-class kafka.tools.GetOffsetShell --broker-list $BOOTSTRAP_SERVERS --topic metrics_test --time -1
kafka-run-class kafka.tools.GetOffsetShell --broker-list $BOOTSTRAP_SERVERS --topic metrics_test --time -2
