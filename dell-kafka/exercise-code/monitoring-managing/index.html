Metrics Layout / Categories / Reference

Kafka metrics have a hierarchical layout. Hundreds or thousands of metrics are available. We provide links to metrics references, Apache and Cloudera.

  • Apache metrics reference includes "client" metrics which are not tracked by Cloudera Manager

  • Cloudera Manager includes additional monitoring, health checks for Kafka

Metrics Reference at Apache.org

Main link for version of Apache Kafka in CDK 3.1.0: https://kafka.apache.org/10/documentation.html#monitoring

Metrics are divided into several sections:

  • Server Metrics: https://kafka.apache.org/10/documentation.html#monitoring

  • Common monitoring Metrics for producer/consumer/connect/streams https://kafka.apache.org/10/documentation.html#selector_monitoring

  • Per broker metrics for producer/consumer/connect/streams Good to know that metrics are provided in /total/ and per-broker (example below) https://kafka.apache.org/10/documentation.html#common_node_monitoring

  • Metrics specific to Producers: https://kafka.apache.org/10/documentation.html#producer_monitoring and https://kafka.apache.org/10/documentation.html#producer_sender_monitoring (This section has another "sub group" for metrics-by-topic - example below)

  • Consumer Group Metrics https://kafka.apache.org/10/documentation.html#new_consumer_group_monitoring

  • Consumer Fetch Metrics https://kafka.apache.org/10/documentation.html#new_consumer_fetch_monitoring (has a "sub group" of metrics per topic, similar to producer metrics by topic)

Metrics Reference at Cloudera

Metrics are listed and described here:

https://www.cloudera.com/documentation/kafka/latest/topics/kafka_metrics.html

They're categorized into:

  • Kafka Metrics
  • Kafka Broker Metrics
  • Kafka Broker Topic Metrics
  • Kafka Mirror Maker Metrics
  • Kafka Replica Metrics (Important this refers to replication of partitions, /not/ specific to Mirror Maker.) Even if you're not using MirrorMaker, this list of metrics has important items to monitor.

Demo: Metrics aggregation on clients

As a demonstration of the layout of "client" metrics, we will run the following:

Create generic topics:

kafka-topics --zookeeper $ZK_CONNECT_STRING \
             --create \
             --topic metrics_test \
             --replication-factor 3 \
             --partitions 3

kafka-topics --zookeeper $ZK_CONNECT_STRING \
             --create \
             --topic metrics_test2 \
             --replication-factor 3 \
             --partitions 3

kafka-topics --zookeeper $ZK_CONNECT_STRING \
             --create \
             --topic metrics_test3 \
             --replication-factor 3 \
             --partitions 3

Run kafka-producer-perf-test on one of the the above topics:

kafka-producer-perf-test \
--num-records 10000 \
--throughput 1000 \
--producer-props bootstrap.servers=$BOOTSTRAP_SERVERS \
--record-size 1000 \
--topic metrics_test \
--print-metrics | tee producerMetrics.log

Note the metrics that are output to the screen and to the file producerMetrics.log

The metrics are laid out according to the specification at https://kafka.apache.org/10/documentation.html#producer_monitoring

For example:

producer-metrics:outgoing-byte-rate:{client-id=producer-1}                           : 266340.395
producer-node-metrics:outgoing-byte-rate:{client-id=producer-1, node-id=node--2}     : 1.676
producer-node-metrics:outgoing-byte-rate:{client-id=producer-1, node-id=node--3}     : 0.600
producer-node-metrics:outgoing-byte-rate:{client-id=producer-1, node-id=node-7}      : 266604.857

Optional: View consumer metrics

Linux $RANDOM command is used to induce read "from-beginning" behavior by specifying a "new" consumer group each time.

kafka-consumer-perf-test \
--broker-list $BOOTSTRAP_SERVERS \
--messages 100000 \
--topic metrics_test \
--group $RANDOM \
--print-metrics \
--show-detailed-stats

Consumer metrics follow a similar pattern (However, there is an odditiy in the kafka-consumer-perf-test implementation which doesn't report the "common" metrics between Producer/Consumer).

However, in later exercises/demos you will see that common metrics are indeed collected by the KafkaConsumer class.

consumer-fetch-manager-metrics:bytes-consumed-rate:{client-id=consumer-1, topic=metrics_test}     : 3281407.525
consumer-fetch-manager-metrics:bytes-consumed-rate:{client-id=consumer-1}                         : 3281300.813
consumer-fetch-manager-metrics:bytes-consumed-total:{client-id=consumer-1, topic=metrics_test}    : 100900000.000
consumer-fetch-manager-metrics:bytes-consumed-total:{client-id=consumer-1}                        : 100900000.000
consumer-fetch-manager-metrics:fetch-latency-avg:{client-id=consumer-1}                           : 13.859
consumer-fetch-manager-metrics:fetch-latency-max:{client-id=consumer-1}                           : 36.000

Examples of Kafka-specific TSQueries and how-to

We will set up some topics, run producers / consumers and then view metrics in Cloudera Manager.

Setup:

Run kafka-producer-performance and kafka-consumer-performance in separate shells;

kafka-producer-perf-test \
--num-records 100000 \
--throughput 10000 \
--producer-props bootstrap.servers=$BOOTSTRAP_SERVERS \
--record-size 1000 \
--topic metrics_test \
--print-metrics

kafka-producer-perf-test \
--num-records 100000 \
--throughput 5000 \
--producer-props bootstrap.servers=$BOOTSTRAP_SERVERS \
--record-size 1000 \
--topic metrics_test2 \
--print-metrics

kafka-consumer-perf-test --broker-list $BOOTSTRAP_SERVERS \
    --topic metrics_test \
    --num-fetch-threads 1 \
    --group $RANDOM \
    --messages 100000 \
    --print-metrics \
    --reporting-interval 10000 \
    --show-detailed-stats

kafka-consumer-perf-test --broker-list $BOOTSTRAP_SERVERS \
    --topic metrics_test2 \
    --num-fetch-threads 2 \
    --group $RANDOM \
    --messages 100000 \
    --print-metrics \
    --reporting-interval 10000 \
    --show-detailed-stats

Commandline tools for monitoring

Recall there's the kafka-consumer-groups command for monitoring lag in consumer groups.

Broker ID to hostname mapping

You can find hostnames for the leader/followers by using the kafka-api-versions command:

kafka-broker-api-versions --bootstrap-server $BOOTSTRAP_SERVERS | grep id

Check offsets for a topic

  • Check earliest and latest offsets for a topic:

The numbers specified by --time -1 and -2 mean "latest offset" and "earliest offset", respectively

kafka-run-class kafka.tools.GetOffsetShell --broker-list $BOOTSTRAP_SERVERS --topic metrics_test --time -1
kafka-run-class kafka.tools.GetOffsetShell --broker-list $BOOTSTRAP_SERVERS --topic metrics_test --time -2
