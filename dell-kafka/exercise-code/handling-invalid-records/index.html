handling-invalid-records

Description

Files related to the "Handling Invalid Records" exercise.

Background

Producers can send faulty records from time to time.

Consumers may be "black box" applications which are difficult to change or configure to handle faulty records. In this exercise we will show how to manage the offsets which are being read by kafka consumer groups. This allows administrators and operations teams to basically "skip past" bad records which are causing Kafka consumers to fail.

Review the Producer and Consumer code

  • The producer is creating comma-separated values, but creates a "faulty" message at random (approximately 10% chance). The exercise code produces a faulty message only once, simulating a corrupt record.

  • The consumer expects four comma-separated fields, and has no error handling :), thus it will throw an exception if it encounters values with less than the expected number of fields.

  • Discussion point: The consumer is committing offsets synchronously, after every message is parsed successfully. If consumers are committing offsets asynchronously and using the "built-in" enable.auto.commit (ConsumerConfig.ENABLE_AUTO_COMMIT_CONFIG in the Java API) it might be possible to simply restart the consumer. However, valid messages may be dropped in this case!

    Imagine this scenario:

 1) Your consumer reads messages 1 .. 50 in partition 0
 2) Consumer's commit-thread tells Kafka that it's successfully processed 1 ..
    50 in partition 0
 3) Your consumer throws an exception on message #1.  When you restart the consumer
    (or the consumer group assigns partition 0 to another group member, the group member
    starts at offset #51

Your thoughts?

  • Note the consumer group ID is called "handling-invalid-records"

Create the Topic

We'll use a new topic (customers) in this exercise, so create it now using the command below:

kafka-topics \
    --zookeeper $ZK_CONNECT_STRING \
    --create \
    --topic customers \
    --replication-factor 1 \
    --partitions 1

Build and Package the Code

mvn clean package

Run the Consumer

Note: You might see messages appear without seeing a crash. This may happen if you have ran this exercise already.

mvn exec:java \
    -Dexec.mainClass="com.cloudera.training.kafka.SimpleConsumer" \
    -Dexec.args="$BOOTSTRAP_SERVERS customers"

Run the Producer

Open a new terminal window, and then run the command below to start a producer that will send a continuous stream of messages to the topic, and will send one "bad" message:

mvn exec:java \
    -Dexec.mainClass="com.cloudera.training.kafka.SimpleProducer" \
    -Dexec.args="$BOOTSTRAP_SERVERS customers"

Use the kafka-consumer-groups utility to view status

The kafka-consumer-groups utility can be used to view the status of a consumer group (the offsets and partitions of the topics being read).

kafka-consumer-groups --bootstrap-server $BOOTSTRAP_SERVERS \
                      --group handling-invalid-records \
                      --describe

You should see output resembling the following:

TOPIC                          PARTITION  CURRENT-OFFSET  LOG-END-OFFSET  LAG        CONSUMER-ID  HOST CLIENT-ID
customers                      0          2575            2732            157        -            -    -

Use kafka-consumer-groups to change the offset

The below command will show what it intends to do by specifying no "execution" argument (e.g. "--execute"). This allows us to double-check the changes we want to make. Remember to change the offset to match the offset you wish your consumer group to "skip" to!

kafka-consumer-groups --bootstrap-server $BOOTSTRAP_SERVERS \
                      --group handling-invalid-records \
                      --topic customers:0 \
                      --reset-offsets \
                      --to-offset 2576

You should see output resembling:

TOPIC                          PARTITION  NEW-OFFSET
customers                      0          13304

Another option of kafka-consumer-groups command allows us to specify --export. This will print a CSV "file" to standard out. We can then save this CSV file to prove that we reset a topic/partition to a specific offset. We could also use the CSV file to perform the action. (See documentation)

kafka-consumer-groups --bootstrap-server $BOOTSTRAP_SERVERS \
                      --group handling-invalid-records \
                      --topic customers:0 \
                      --reset-offsets \
                      --to-offset 2576 \
                      --export

Finally, specify --execute to actually change the offset:

kafka-consumer-groups --bootstrap-server $BOOTSTRAP_SERVERS \
                      --group handling-invalid-records \
                      --topic customers:0 \
                      --reset-offsets \
                      --to-offset 2576 \
                      --execute

Rerun your consumer

Re-run the consumer. Messages will be printed quickly to the screen, but try to catch the first message printed. The offset should be the offset which you specified in the kafka-consumer-groups command above.

mvn exec:java \
    -Dexec.mainClass="com.cloudera.training.kafka.SimpleConsumer" \
    -Dexec.args="$BOOTSTRAP_SERVERS customers"

Ways to deal with corrupt / invalid / incompatible ("bad") records

  • All three problems can cause crashes in consumers

  • If consumers retry reading the same bad message (e.g. Consumer Groups) the same bad message(s) will be read by other consumer processes, causing the same problem.

Handling bad messages in consumers:

Best practices:

  • Log message content and detailed error codes
  • Log the partition and offset of bad messages
  • Design your (consumer) application to handle bad messages
    • Skip bad messages?
    • Stop processing?
    • Attempt to repair? (e.g. use some kind of default value?)

If you have no control over consumer behavior

Options for handling bad messages

  • Update the consumer group's message offsets to "skip" the offending message

    • Requires partition ID and offset

    • NOTE: "Bad" messages may already be skipped by consumers that commit offsets asynchronously. Consider the scenario below:

12:00 read messages 50-100 from partition 1
12:01 auto-commit tells Kafka "I've processed message 100 in partition 1"
12:02 when processing message 50, application crashes
12:03 Consumer group may continue to function, because consumer group assigns partition 1
     to another consumer, which begins reading at message 100 from partition 1
