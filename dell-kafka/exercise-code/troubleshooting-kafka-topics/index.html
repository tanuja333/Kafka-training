Troubleshooting Kafka Topics

In the next series of exercises, a program called "TrogdorJunior" will be ran. It will perform various operations on your Kafka cluster. It's up to you to identify and troubleshoot the problems. Each exercise has a "hints" section, under "Hints" below.

Note: The program is meant to be difficult to read, and does not output a lot of information. When you run TrogdorJunior, it will not give out a lot of information. Go to the cluster after running the program, and start diagnosing problems.

Hints

Commandline tools for monitoring

Recall there's the kafka-consumer-groups command for monitoring lag in consumer groups.

Useful command: kafka-topics

The kafka-topics command is useful for describing a topic and viewing partition assignment. Use the --describe switch to describe a topic.

Broker ID to hostname mapping

You can find hostnames for the leader/followers by using the kafka-api-versions command:

kafka-broker-api-versions --bootstrap-server $BOOTSTRAP_SERVERS | grep id

Check offsets for a topic

  • Check earliest and latest offsets for a topic:

The numbers specified by --time -1 and -2 mean "latest offset" and "earliest offset", respectively

kafka-run-class kafka.tools.GetOffsetShell --broker-list $BOOTSTRAP_SERVERS --topic metrics_test --time -1
kafka-run-class kafka.tools.GetOffsetShell --broker-list $BOOTSTRAP_SERVERS --topic metrics_test --time -2

Exercise: Trogdor Junior Exercise 1

You can run Trogdor Junior and specify a number, indicating which "test" will be ran.

The program might only send / receive data for a specified time or number of records. If you identify/fix a problem and wish to simply re-run the data-ingest or data-consumption, then specify the data_only flag as the last parameter (examples below).

Run Trogdor Junior Test 1

Run the command below, and identify / troubleshoot problems. This test will create and write data to the strongbad topic.

mvn clean package
mvn -e exec:java \
    -Dexec.mainClass="com.cloudera.training.kafka.TrogdorJunior" \
    -Dexec.args="$BOOTSTRAP_SERVERS 1 strongbad"

Try to find "the problem". Use the Hints sections above (and the More Hints section below).

If the TrogdorJunior program has quit producing/consuming messages, you can re-run it by specifying a data_only parameter:

mvn clean package
mvn -e exec:java \
    -Dexec.mainClass="com.cloudera.training.kafka.TrogdorJunior" \
    -Dexec.args="$BOOTSTRAP_SERVERS 1 strongbad data_only"

You will not see much output. Go identify the problem.

Fix for Exercise 2

The messages received rate should show messages received for only one broker. Also, the kafka-topics --describe command would show that only one broker is the leader for all six of the drevil topic.

Run the rebalance tool. For more information, see:

Partition Reassignment

Create a file topics.json, with YOUR_TOPIC_HERE replaced by the topic you wish to rebalance:

{"topics": [ {"topic":"YOUR_TOPIC_HERE"} ], "version": 1}

Create a rebalance "plan" by running the kafka-ressign-partitions tool with the --generate option. This is also useful for seeing the current partition assignments for the topic(s) you specified. Remember to replace YOUR_BROKER_LIST below with a comma-separated list of broker IDs.

kafka-reassign-partitions --zookeeper $ZK_CONNECT_STRING \
    --topics-to-move-json-file topics.json \
    --broker-list "YOUR_BROKER_LIST"  \
    --generate

You will see both the current assignment and the "proposed" assignment. For an easy attempt, highlight the "proposed" assignment, and paste it into a new file, perhaps proposed.json. If you want to alter the proposed assignment, it might be worthwhile to format the JSON using the following command:

python -m json.tool proposed.json > pretty_proposed.json

Run the rebalancer

kafka-reassign-partitions --zookeeper $ZK_CONNECT_STRING \
    --execute \
        --reassignment-json-file pretty_proposed.json

You can check the status by running:

kafka-reassign-partitions --zookeeper $ZK_CONNECT_STRING \
    --verify \
        --reassignment-json-file pretty_proposed.json

You can view the partition assignments by running the kafka-topics command:

kafka-topics --zookeeper $ZK_CONNECT_STRING
             --describe 
             --topic YOUR_TOPIC_HERE

You might notice that the partitions / leaders have not yet been assigned according to the partition assignments in the json file.

For example, your JSON file might specify an even balance of leaders between broker 18 and 20. But, when you run kafka-topics --describe, you see this:

Topic:drevil    PartitionCount:6        ReplicationFactor:2     Configs:
Topic: drevil   Partition: 0    Leader: 20      Replicas: 18,20 Isr: 20,18
Topic: drevil   Partition: 1    Leader: 20      Replicas: 20,18 Isr: 20,18
Topic: drevil   Partition: 2    Leader: 20      Replicas: 18,20 Isr: 18,20
Topic: drevil   Partition: 3    Leader: 20      Replicas: 20,18 Isr: 18,20
Topic: drevil   Partition: 4    Leader: 20      Replicas: 18,20 Isr: 20,18
Topic: drevil   Partition: 5    Leader: 18      Replicas: 20,18 Isr: 20,18

If you wait awhile, you should see the partitions / leaders become assigned. This is due to the "preferred leader election" process needing to be ran. Note that on the first line, "replicas" is 18,20 and the leader is 20. This should soon change to Leader: 18 Replicas 18,20
