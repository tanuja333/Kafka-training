isr-versus-acks

Comparing ISRs (In-Sync Replicas) and ACKs

This demo/exercise is meant to illustrate how producers use request.required.acks and a topics min.insync.replicas properties.

Producers tell a leader how many ACKs are required for a message to be considered "committed". This can provide lower latency, but at the risk that messages are lost if a leader goes down.

Producers can specify the following settings for acks as follows:

  • -1 or "all" : A message must be acknowledged by all in-sync replicas
  • 1 : Only the leader need acknowledge the message
  • 0 : No acknowledgement necessary ("Fire and forget")

https://kafka.apache.org/10/javadoc/index.html?org/apache/kafka/clients/producer/KafkaProducer.html

Keep in mind that brokers uses a /fetch/ mechanism (not a /push/ mechanism) to replicate messages. The leader does not attempt to push replicas to followers, the followers poll the leader, so the leader puts a message in a queue

Take this scenario:

  1. Producer sends a message with ACKS=1
  2. Leader receives message, acknowledges it, marks it as committed
  3. Consumer fetches message from the leader
  4. Leader goes down
  5. Failover occurs and consumer comes back for the same message

Create a Topic with ISRs = replication factor = brokers

This is not necessarily a recommended approach; The number of min.insync.replicas typically is smaller than the replication-factor.

kafka-topics --zookeeper $ZK_CONNECT_STRING \
             --create \
             --topic not_enough_isrs \
             --replication-factor 3 \
             --config min.insync.replicas=3 \
             --partitions 1

Start a Consumer

kafka-console-consumer \
    --bootstrap-server $BOOTSTRAP_SERVERS \
    --topic not_enough_isrs

Start a producer with required ACKs = 1

export KAFKA_OPTS="-Dlog4j.configuration=file:log4j.properties.trace -DthisErrorLog=prodAckONE.log"
kafka-console-producer --broker-list $BOOTSTRAP_SERVERS \
    --topic not_enough_isrs \
    --sync \
    --request-timeout-ms=10000 \
    --request-required-acks 1

Push messages from this console. For clarity, produce messages beginning with "1", such as:

1 a
1 b
1 c

Leave this process running.

Start a Producer with required ACKs = -1

export KAFKA_OPTS="-Dlog4j.configuration=file:log4j.properties.trace -DthisErrorLog=prodAckALL.log"
kafka-console-producer --bootstrap-servers $BOOTSTRAP_SERVERS \
    --topic not_enough_isrs \
    --request-required-acks -1 \
    --request-timeout-ms=10000 \
    --sync

Push messages to this console. For clarity, prefix messages with "all", such as:

all a all b all c

Leave this process running.

Kill or Pause One of the Brokers

Note the Broker IDs for the leader and the value of "ISRs" in the output from the kafka-topics command:

kafka-topics --zookeeper $ZK_CONNECT_STRING --describe --topic not_enough_isrs

We will now stop one of the brokers.

  • You can use Cloudera Manager to stop one of the brokers (easiest). Proceed to Cloudera Manager and simply stop one of the brokers. Then go to the section titled "Try the kafka-topics command again".

  • Or you can SSH to one of the brokers and "pause" a broker. If you want to use the commandline, follow the steps in the next subsection.

Pausing Broker using SSH on the Command Line

You can find hostnames for the leader/followers by using the kafka-api-versions command:

kafka-broker-api-versions --bootstrap-server $BOOTSTRAP_SERVERS | grep id
  1. SSH to one of the brokers using root@
  2. Find the PID for Kafka broker using a command such as jps
  3. Use the following command to "pause" the broker:
kill -s SIGSTOP <PID>

After you have paused the broker, proceed to the next step. Don't forget to "un-pause" the broker with this command after this demo/exercise.

# Un-pause the broker
kill -s SIGCONT <PID>

Try the kafka-topics Command Again

What difference do you see from the first time you ran the topics command?

kafka-topics --zookeeper $ZK_CONNECT_STRING --describe --topic not_enough_isrs

Produce More Messages

In both the producers you've started above, type more messages into the console. Do you notice a difference in the consumer?

Check the log files for each producer. Review the prodAckALL.log file and look for "retrying", "NotEnoughReplicasException", and NOT_ENOUGH_REPLICAS.

Review the prodAckONE.log log file and look for similar messages.

Cloudera Manager Alerts

If you check Cloudera Manager, you will see "Bad Health" for the Kafka Service. You should see warnings for "Lagging Replicas Tests".

Bring Broker Back Up

Using Cloudera Manager or the "unpause" command:

# Un-pause the broker
kill -s SIGCONT <PID>

Review Logs Using CM

After you bring up the brokers, check the Kakfa logs in Cloudera Manager.

  1. Go to the "Diagnostics" drop-down at the top of the screen and choose "logs"
  2. In the "Sources" drop-down, select only "Kafka"
  3. Search for not_enough_isr and not_enough_isr.*ISR

You should see log messages regarding:

  • ISRs shrinking from 3 to 2 to 1 replicas ("[Partition not_enough_isrs-0 broker=9] Shrinking ISR from 9,11,7 to 9,7)

  • ISRs increasing from 1 to 2 to 3 replicas after you restarted the brokers: ("[Partition not_enough_isrs-0 broker=9] Expanding ISR from 9 to 9,7")

  • Errors regarding NotEnoughReplicasException from the leader
